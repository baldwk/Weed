# External Plugins

This document describes how to build, package, import, and troubleshoot Weed external plugins.

## Runtime Model

External plugins are managed .NET assemblies loaded from the user plugin directory:

```text
%LOCALAPPDATA%\Weed\plugins\<plugin-id>\
  manifest.json
  My.Plugin.dll
  My.Plugin.deps.json
  dependency.dll
  runtimes\
  assets\
```

Weed scans this directory on startup. Each plugin is loaded into its own `AssemblyLoadContext`, and
`Weed.Abstractions` is supplied by the host. Plugin dependencies should live next to the plugin DLL
or be described by the plugin `.deps.json` generated by `dotnet publish`.

External plugins are not hot-reloaded yet. After importing, restart Weed to load the new or updated
plugin. If Windows keeps a loaded plugin DLL or native dependency locked, close Weed before replacing
that plugin.

## Manifest

Every package root must contain `manifest.json`:

```json
{
  "id": "example.plugin",
  "name": "Example Plugin",
  "version": "0.1.0",
  "sdkVersion": "0.1",
  "assembly": "Example.Plugin.dll",
  "entryType": "Example.Plugin.ExamplePlugin",
  "runtime": {
    "resident": false
  },
  "activations": [
    {
      "type": "keyword",
      "keyword": "ex",
      "command": "example.search"
    }
  ],
  "permissions": []
}
```

Important fields:

- `id`: stable plugin id. It becomes the install folder name under `%LOCALAPPDATA%\Weed\plugins`.
- `assembly`: DLL path relative to `manifest.json`.
- `entryType`: public type that implements `IWeedPlugin`.
- `activations`: supported entry points: `keyword`, `hotkey`, and `implicitQuery`.
- `permissions`: host capabilities the plugin expects. In the current MVP these are displayed for
  diagnostics and review; they are not a security sandbox.

## Development Workflow

Use `templates\plugin` for a minimal plugin, or create a class library that references
`Weed.Abstractions`.

Inside this repository, use a project reference:

```xml
<ProjectReference Include="..\..\Weed.Abstractions\Weed.Abstractions.csproj">
  <Private>false</Private>
</ProjectReference>
```

Outside this repository, reference the `Weed.Abstractions.dll` shipped with a Weed build:

```xml
<Reference Include="Weed.Abstractions">
  <HintPath>..\Weed\Weed.Abstractions.dll</HintPath>
  <Private>false</Private>
</Reference>
```

Publish the plugin instead of copying only the DLL:

```powershell
dotnet publish .\Example.Plugin.csproj -c Release -r win-x64 --self-contained false -o .\dist\example.plugin
```

The published folder should contain `manifest.json`, the plugin DLL, the `.deps.json` file, and any
dependency DLLs or runtime assets.

## Import Design

Weed Settings > External Plugins imports local plugin artifacts into the user plugin directory. It does
not require a registry URL. Supported imports:

- a `.zip` package whose root contains `manifest.json`
- a folder whose root contains `manifest.json`
- a `.zip` or folder with a single child directory containing `manifest.json`
- a source folder with `manifest.json` and one plugin `.csproj`; Weed runs `dotnet publish` and imports
  the published output
- a `.dll` file with a matching `manifest.json` next to it
- a `.dll` file that contains a public `IWeedPlugin` implementation; if the plugin type exposes a
  static `Manifest` property or field, Weed uses it, otherwise it generates a minimal manifest

The importer:

1. Reads and validates `manifest.json`.
2. Verifies `id`, `assembly`, and `entryType`.
3. Verifies the assembly path stays inside the package and exists.
4. Copies the package to `%LOCALAPPDATA%\Weed\plugins\<manifest.id>`.
5. Optionally replaces an existing plugin folder with the same id.
6. Leaves the plugin unloaded until the next Weed restart.

For source folders, the importer publishes with the current Windows runtime identifier, Release
configuration, and `--self-contained false`. If multiple `.csproj` files are present, name the plugin
project after the manifest `assembly` value or import the project folder directly.

Manual import is also supported:

```powershell
$target = "$env:LOCALAPPDATA\Weed\plugins\example.plugin"
New-Item -ItemType Directory -Force -Path $target
Copy-Item .\dist\example.plugin\* $target -Recurse -Force
```

Then restart Weed.

## Independent Repository Publishing

Third-party plugins should be published from their own GitHub repositories. A recommended repository
layout is:

```text
weed-plugin-example/
  src/
  manifest.json
  README.md
  .github/workflows/release.yml
```

On each version tag, publish with:

```powershell
dotnet publish .\src\Example.Plugin.csproj -c Release -r win-x64 --self-contained false -o .\dist\example.plugin
Compress-Archive -Path .\dist\example.plugin\* -DestinationPath .\example.plugin-0.1.0-win-x64.zip
```

The GitHub Release should upload:

- `<plugin-id>-<version>-win-x64.zip`
- optional release notes or checksums for users who want to verify downloads

To publish a checksum:

```powershell
(Get-FileHash -Algorithm SHA256 .\example.plugin-0.1.0-win-x64.zip).Hash.ToLowerInvariant()
```

Users install by downloading the ZIP, DLL, or source folder and choosing Import in Settings >
External Plugins. External plugins are normal in-process plugins. Only import plugins you trust;
declared permissions are descriptive and are not a sandbox.

## OCR External Plugin

`External Plugins\Weed.Plugins.Ocr` is an example of a non-built-in external plugin. `Weed.App` does not
reference it; the plugin is packaged and imported like third-party plugins.

Build:

```powershell
dotnet build "External Plugins\Weed.Plugins.Ocr\Weed.Plugins.Ocr.csproj"
```

Package with PP-OCRv5 Chinese models:

```powershell
powershell -ExecutionPolicy Bypass -File scripts\package-ocr-plugin.ps1 -FetchModels
```

Output:

```text
artifacts\plugins\weed.ocr\
artifacts\plugins\weed.ocr.zip
artifacts\plugins\weed.ocr-0.1.0-win-x64.zip
artifacts\plugins\weed.ocr-0.1.0.plugin-release.json
```

Current size:

- OCR model files: about 21 MiB.
- ZIP package with RapidOCR/ONNX/OpenCV runtime and models: about 60 MiB on this machine.

Usage after import and restart:

- `ocr`: show OCR actions.
- `ocr "C:\path\image.png"`: recognize an image file.
- `Shift+Alt+O`: capture a region and open the OCR result in the launcher.

The OCR plugin does not read from the clipboard. Its default result action copies recognized text to
the clipboard. Opening a `.txt` file under the plugin data directory remains available as a secondary
action.

## Troubleshooting

- If a plugin does not appear, restart Weed and check Settings > plugin > Advanced > Log Tail.
- If dependencies fail to load, package with `dotnet publish`, not `dotnet build`.
- If a plugin ZIP imports but does not load, verify `manifest.json` is at the package root and
  `assembly` points to the published DLL.
- Install only trusted plugins. External plugin code runs inside the Weed process.
